k8s☞06无状态服务deployment-pod更新

zyh 2020-09-14 14:44:04
容器 > k8s

前言

对于更新,我们一般会采用滚动更新,即:

  • 先扩展一部分新版本
  • 删除一部分老版本
  • 重复上述行为,直至所有老版本被替换.

对于回滚,我们想回滚到任何一个之前的版本.

而deployment对象可以帮我们实现这些.

https://kubernetes.io/docs/concepts/workloads/controllers/deployment/

一个基本的deployment例子

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
        ports:
        - containerPort: 80

例子需要注意的点：

  • metadata： 这里会定义name，labels，namespace，annotations。其中 annotations （注解）是特殊的，你可以用它来定义一些工具集，例如在阿里云的k8s服务中，你可以添加镜像快照注解信息，从而让 pod 无需从镜像仓库下载启动，而是直接通过快照加速启动。
  • selector： deployment使用selector指定的label关联pod。

构建deployment

创建一个nginx的deployment

kubectl apply -f nginx-dep.yaml
===
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-dep
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx
        ports:
        - containerPort: 80

这里dep的配置中spec.template.spec.container.image=nginx，我没有设置具体版本

确认当前版本

查看版本历史，使用kubectl rollout history

这里涉及到 rollout 指令，回滚的时候也会用到.

kubectl rollout history deployment/nginx-dep
#---
deployment.apps/nginx-dep 
REVISION  CHANGE-CAUSE
1         <none>

上述结果中，版本1没有添加 CHANGE-CAUSE 信息

添加注释信息，使用kubectl annotate deployment/<> kubernetes.io/change-cause=“first”

kubectl annotate deployment/nginx-dep kubernetes.io/change-cause="first"
kubectl rollout history deployment/nginx-dep
#---
deployment.apps/nginx-dep 
REVISION  CHANGE-CAUSE
1         first

你也可以在执行命令的时候，追加 --record 来将执行的命令写入到当前change-cause

如果你不小心写错了，你可以通过添加- 来删除注解，就像下面这样，命令最后有一个-

kubectl annotate deployment/nginx-dep <注解key>-

更新版本

发布版本1.19.2，将deployment中所有nginx的image版本从默认变更为1.19.2。

kubectl set image deployment/nginx-dep nginx=nginx:1.19.2
kubectl annotate deployment/nginx-dep kubernetes.io/change-cause="nginx image to 1.19.2"
kubectl rollout history deployment/nginx-dep
#---
deployment.apps/nginx-dep 
REVISION  CHANGE-CAUSE
1         first
2         nginx image to 1.19.2

除了使用set指令修改，你也可以通过edit修改yaml配置的方式进行

kubectl edit deployment/nginx-dep

更新状态查看，可运行：

kubectl rollout status deployment/nginx-dep

更深层的rs状态查看，可运行：

kubectl get rs -l app=nginx

-l 是调用标签过滤

再发布一个版本1.18，将deployment中所有nginx的image版本从1.19.2变更为1.18。

kubectl set image deployment/nginx-dep nginx=nginx:1.18
kubectl annotate deployment/nginx-dep kubernetes.io/change-cause="nginx image to 1.18"
kubectl rollout history deployment/nginx-dep
#---
deployment.apps/nginx-dep 
REVISION  CHANGE-CAUSE
1         first
2         nginx image to 1.19.2
3         nginx image to 1.18

这样,我们总共有三个版本，你可以通过追加版本号来看具体的版本内容

kubectl rollout history deployment/nginx-dep --revision=<num>

回滚版本

我们选择将nginx版本回滚到first第一个版本.

kubectl rollout undo deployment/nginx-dep --to-revision=1
kubectl rollout history deployment/nginx-dep
#---
deployment.apps/nginx-dep 
REVISION  CHANGE-CAUSE
2         nginx image to 1.19.2
3         nginx image to 1.18
4         first

更新策略

deployment 的 .spec 中可以添加一些策略.

spec:  
  minReadySeconds: 3  # pod 就绪时间，在此时间之前，deployment 认为 pod 还没有准备好. 默认0
  revisionHistoryLimit: 10 # 最大版本保留次数. 默认10 
  strategy:
    type: rollingUpdate # 定义变更策略. 除了 rollingUpdate，还可以是Recreate.Recreate指的是先删除所有pod,再创建.
    rollingUpdate:
      maxUnavailable: 30% # 变更期间最多存在多少个不可用的pod,可以是具体数字
      maxSurge: 30% # 变更期间最多存在多少个超出已定义副本的pod数量,可以是具体数字

maxUnavailable 和 maxSurge 设置相等即可. 即开多少个新的，就关多少个老的

最终样子

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-dep
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  minReadySeconds: 3  # pod 就绪时间，在此时间之前，deployment 认为 pod 还没有准备好. 默认0
  revisionHistoryLimit: 10 # 最大版本保留次数. 默认10 
  strategy:
    type: rollingUpdate # 定义变更策略. 除了 rollingUpdate，还可以是Recreate.Recreate指的是先删除所有pod,再创建.
    rollingUpdate:
      maxUnavailable: 30% # 变更期间最多存在多少个不可用的pod,可以是具体数字
      maxSurge: 30% # 变更期间最多存在多少个超出已定义副本的pod数量,可以是具体数字
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx
        ports:
        - containerPort: 80

暂停服务

kubectl rollout pause deployment/nginx-dep

启用暂停后，再启用更新，则新的更新在暂停期间不会产生任何效果

恢复服务

kubectl rollout resume deployment/nginx-dep

启用恢复后，则暂停期间启用的更新将开始发挥作用

小结

kubectl rollout history deployment/<> # 查看对象历史
kubectl rollout undo deployment/<> --to-revision=<> # 对象版本回退
kubectl rollout status deployment/<># 展示执行状态
kubectl rollout pause deployment/<> # 暂停此次版本操作行为
kubectl rollout resume deployment/<> # 恢复此次版本操作行为
kubectl set image # 修改镜像配置
kubectl annotate # 添加一个注释

更新过程：

  • 创建新的replicaset
  • 将新的replicaset扩展
  • 将旧的replicaset缩容
  • 新的replicaset对应的pod已就绪或可用

当kubectl rollout status deployment/<>命令执行完后，$? 状态为 0，则说明 deployment 处于 complete。
